## What

Build the `/work` route and the three case-study routes — the entire "Work" half of the portfolio. After this phase, every `SelectedWork` row on the home page resolves to a real destination, the nav `02 / Work` link stops 404-ing, and the three flagship case studies (AvsB, Kemon Doctor, Client Experimentation Work) read as fully-authored MDX content rather than placeholders.

Phase 6 lands **four new routes**, **two new top-level work primitives** (`Filters`, `IndexRow`), **seven case-study primitives** (`CaseHero`, `CaseMeta`, `CaseBlock`, `CaseVisuals`, `CodeMock`, `NextCase`, `Countries`), **one shared layout** (`CaseLayout`), **a global MDX-components map extension** so case MDX can use the primitives without per-file imports, and an **extended `WorkProject` type / `workProjects` array** that grows from today's 3 entries to the full 6-project catalogue the index demands.

## Why

Phases 1–5 stood up the project, the tokens, the chrome, the home, About, and Contact. The home page tells a visitor what Mainul does in twelve seconds — and three rows in, it teases three case studies. Today, **all three of those rows 404.** Phase 6 turns the teaser into a deliverable. From a credibility standpoint, this is the most important page on the site after the home — a frontend developer specialising in A/B testing whose case-study page returns "page not found" is dead on arrival.

A second motivation: **MDX is the format the masterplan and scaffold chose for case studies**, but no MDX has been authored yet. The wiring exists (`next.config.ts` + `mdx-components.tsx` + `@next/mdx` already installed), but it's untested in production. Phase 6 stress-tests the pattern: frontmatter parsing, layout wrapping, globally-injected MDX components, image overrides, and code-block styling all get exercised on the first real content. Better to fight any MDX rough edges now (one phase) than during a launch push.

A third: **the `WorkProject` type and `workProjects` array were always thin** — they cover the 3 home-featured entries with a tiny field set (`slug`, `num`, `title`, `summary`, `tags`, `featured`, `order`, `year?`, `type?`). The index needs categories for filtering, a shorter per-row summary, and a `hasCase` flag so the stub rows know they don't link anywhere new. Extending the type once, now, is cleaner than extending it three times across Phase 6, Phase 7, and a hypothetical Phase 8.

## Why now (not later)

- **It's the next phase on the masterplan**, and it unblocks the home page's most visible broken interaction (the `SelectedWork` 404s tracked at `futureWorks.md:15`).
- **The Phase 7 polish list assumes case pages exist** — OG images use the case heroes as their visual reference, the sitemap covers all 7 routes, and the Lighthouse pass is on all 7. Doing Phase 7 before Phase 6 would mean either polishing pages that don't exist or splitting Phase 7 in two.
- **MDX is a one-time investment.** Once Phase 6 lands the `mdx-components.tsx` map and the `CaseLayout` wrapping pattern, Phase 7 doesn't need to touch any MDX wiring. Putting MDX work off until Phase 7 would force Phase 7 to do both the polish _and_ the MDX foundation, which is a worse package.
- **The `WorkProject` extension is cheaper now than later.** Two consumers (home + work index). Adding `categories` / `hasCase` / `metaShort` later would mean either bolting them onto rows already in production or doing a refactor PR. Doing it in the same PR that adds the rows is the right shape.
- **The 320px guardrail** (`futureWorks.md:11`) needs case-page testing — the case display titles (`clamp(8.4rem, 17vw, 27rem)`) are the most aggressive type sizes on the site. Confirming they survive on iPhone SE 1st-gen on a real page is something only Phase 6 can do.

- Per-stub-project case pages (Phase 8+ at earliest, only if content materialises).
- Real OG images per case (Phase 7).
- The Phase-7 polish list (sitemap, robots, full Metadata API beyond `title` + `description`, favicon, self-hosted CV).
- Animation polish beyond what the static template does (no Framer Motion, no GSAP — see masterplan §11).
- A search field on `/work`. Six rows; filters are enough.
- Pagination on `/work`. Six rows; no point.
- Tag-based search beyond the 4 filter chips. (Tag chips on a row are visual, not interactive.)
- Any new top-level dependency. MDX, Zod, and the existing primitives cover everything.
- Dark mode.
- Any analytics, tracking, or third-party widget.

## Risk

- **MDX + `additionalData` interaction.** `futureWorks.md:14` notes that `sassOptions.additionalData` isn't being applied to component `*.module.scss` modules under Next 16.2.6 / Turbopack — Phase 3 worked around this with explicit `@use 'variables' as *; @use 'mixins' as *;` per file. The case-study modules will follow the same per-file `@use` rule. **Mitigation:** every new `.module.scss` in this phase starts with the explicit `@use` lines, matching Phase 3–5 modules. No new variance introduced.
- **MDX named imports for `frontmatter` are not the most-trodden path.** `@next/mdx` supports `export const` at the top of an MDX file, but the _typescript-correct_ pattern for importing the named export plus the default in a `.tsx` shim is something MDX docs treat as advanced. **Mitigation:** the spec includes a working snippet for the shim, and the very first thing the implementation does (after scaffold) is verify the shim resolves with `npx tsc --noEmit`. If the import shape blows up, fallback is YAML frontmatter via `remark-frontmatter` — one dependency add, documented in the spec as Plan B.
- **Filter state during route transitions.** If a user changes a filter and then clicks a row link, the client island unmounts mid-state. **Mitigation:** the filter state is purely visual (it doesn't persist to URL or localStorage), and the client island is a leaf — unmount is clean. No effect cleanup needed beyond what React does by default.
- **6-row index re-render on every filter change.** Filtered render means a `.map` over the filtered subset on each click. Six rows; cost is negligible. **Mitigation:** none needed — flagging here only to head off the temptation to add `useMemo` ceremony.
- **`CaseMeta` per-case label variation (`Span` / `Reach` on Client vs `Year` / `Status` on the others).** Easy to accidentally lock the labels as constants in the component instead of accepting them per case. **Mitigation:** the type signature for `CaseMeta` takes a `cells: readonly { label: string; value: string; accentDot?: boolean }[]` prop — labels are data, not enum.
- **Visual placeholder treatment must read as intentional.** Striped fill + mono caption is what the legacy ships. There's a risk that without real screenshots, the case pages feel hollow. **Mitigation:** treat the placeholder as a design element — caption clearly says `Dashboard · in-progress` / `Search · mobile` so readers know they're not looking at a broken image. A `futureWorks.md` entry tracks "real case screenshots" so they don't fall off the radar at launch.
- **320px guardrail on case heroes.** `clamp(8.4rem, 17vw, 27rem)` floors at 8.4rem ≈ 84px and the AvsB hero ("Experiments,") could overflow on a 320px iPhone SE 1st-gen viewport. **Mitigation:** verify each case hero at 320px during the verify step. If overflow appears, add a ≤380px breakpoint that drops the floor to ~6.4rem on case heroes specifically (display-lg sizing).
- **`/work/client`'s "Outcomes" block is intentionally empty.** A reader expecting numbers will find a confidentiality note instead. **Mitigation:** the copy is unchanged from the legacy and already addresses the expectation directly ("Honesty is the point of this section. Real numbers land here only once a client signs off on sharing them."). Don't soften.
- **`NextCase` wrap behaviour** — Client's next-case loops back to AvsB with a `Back to · 01` label, not `Next case · 04`. A future change adding a 4th case would silently break the wrap label. **Mitigation:** `case-registry.ts` is the single source of truth for the next-case mapping AND the label; adding a 4th case is a single-file edit there + one new MDX folder, not a hunt across components.
- **Lighthouse a11y target = 100.** Long-form MDX bodies + interactive filter row + multiple `Reveal` instances each carry their own a11y considerations. **Mitigation:** keyboard pass + VoiceOver pass before merging; same bar as Phase 5.

## What I think you might want to push back on

- **The "page.tsx shim + content.mdx" split for case routes.** The cleaner-looking alternative is a single `page.mdx` at the route, with the shim work done invisibly via a wrapper. The honest reason for the split: App Router's MDX support gives `useMDXComponents` no way to receive structured frontmatter, and YAML frontmatter requires a remark plugin. If you'd rather take the YAML-plus-plugin route (one new dev dependency: `remark-frontmatter`), the spec can flip to a single-file-per-case structure. I default to the shim because it keeps the dep tree the same and the typescript types tight, but the cleaner author ergonomics of YAML-in-MDX are real.
- **Filters as a client island vs. URL-state filters.** I default to local component state. The case for URL-state (`/work?cat=product`) is that a filter selection becomes shareable and survives back/forward navigation. Cost: a small bit of router plumbing. If you want the URL-state, easy switch, but it crosses a line into "this filter is a feature, not a chip" — for a 6-row list, that feels like over-investment.
- **Stub rows linking to `/work/client` (no anchors).** If you'd rather generate three minimal stub pages (`/work/flatwhite`, `/work/crypto-gods`, `/work/pascal`) right now — each one paragraph saying "Confidential; ask me about it directly." — I'm happy to do that and it would be cleaner navigation. The cost is six more files for content that's deliberately thin. I default to "no" because masterplan §13 defaults to no, and because three stubs all pointing at the same Client page is a fair description of where that work sits in the portfolio.
- **CodeMock token colours.** I default to a small hardcoded palette in `_CodeMock.module.scss` (comment grey, prompt accent, keyword blue, value gold, string green). The colours come from the legacy template's inline `<style>` block. If you'd rather the CodeMock honour the runtime `--accent` (so changing accent via the home Tweaks panel ripples into the AvsB terminal too), that's a one-line change in the SCSS module. I defaulted to fixed colours because the AvsB terminal is a _case-study visual asset_ — a designed thing, not part of the live accent surface.
- **`Countries` as an inline JSX component in `content.mdx`** vs. as a frontmatter-driven block. If you'd rather every case's "supporting structured data" (cases, countries, stack lists) live in frontmatter for reasoning consistency, that's a different shape — frontmatter grows, MDX bodies thin. I default to JSX-in-body because Countries is a body element, not a header element. But the consistency argument is fair.
- **Real case screenshots.** Phase 6 ships with striped-fill placeholders. If you have any case screenshots ready (especially the AvsB dashboard and the Kemon Doctor mobile flows), dropping them into `public/work/` and wiring them to `CaseVisuals` adds ~30 minutes to the phase and meaningfully changes the page's feel. I defaulted to placeholders because the legacy ships placeholders and because not having to ship screenshots unblocks Phase 6 entirely.
