<CaseBlock num="01" label="Overview" heading="What it is">
  Radius centralises everything an experimentation team learns about a website and its customers — A/B test results, UXR observations, surveys, competitor analyses, research notes — into a **single, searchable hub** that everyone in the organisation can use.

  Built at **Conversion.com** for their own consulting teams, it's now also opened up to <span className="accent">client stakeholders</span> so they can access their own experiment history and insights directly.
</CaseBlock>

<CaseBlock num="02" label="My role" heading="What I did">
  **Frontend developer on a close team collaboration.** The product strategy, taxonomies, and synthesis methodology came from the wider Conversion team — my work was the frontend that brings it to life.

  - Next.js + TypeScript application surface — pages, routing, data layer
  - SCSS-based styling system shared across the app
  - **GSAP-driven motion** for transitions, list reveals, and modal choreography
  - **Streaming API integration** so large result sets render progressively rather than blocking on a full payload
  - Cross-device QA across the typical surfaces this kind of tool gets used on — laptop, second monitor, the occasional meeting screen
</CaseBlock>

<CaseBlock num="03" label="The problem" heading="What it solves">
  Every insight has an **impact radius** — and without easy access to it, that radius stops at whichever stakeholder meeting or shareable report introduced it.

  Cornell research puts a number on the cost: people spend roughly **5 hours a week** simply searching for information trapped inside tools and documents; data and research professionals lose around **20% of their time duplicating research** that already exists somewhere in the organisation.

  Conversion saw the same pattern internally — findings from one consultant rarely reached the wider org (other consultants, designers, management, sales) without manual hand-holding.
</CaseBlock>

<CaseBlock num="04" label="The approach" heading="Three steps">
  The team's model for expanding the radius:

  - **Organise** — capture experiment results and UXR observations against standardised taxonomies (lever, device, page type, psychological principle, components, executions, methodology) so a future search actually returns what it should.
  - **Synthesise** — turn raw observations from surveys, user testing, heuristic reviews, and heatmaps into actionable insights, not just data dumps.
  - **Democratise** — Radius itself: a searchable, shareable, single-source-of-truth UI anyone in the organisation can use without onboarding.
</CaseBlock>

<CaseBlock num="05" label="What I built" heading="The frontend">
  The parts of the system I owned, end-to-end on the client:

  - **Faceted search index** — filters across type (experiments / insights / observations), outcome (winner / loser / inconclusive), and master lever (trust / cost / usability / comprehension / motivation), plus free-text. The catalogue's superpower.
  - **One-click detail surface** that pulls in the full context of a test or observation — visual, narrative, and numbers — so consumers don't have to chase down the original report.
  - **Save-for-later** so individual users can build a working set of references they care about.
  - **Share-to-stakeholder** via Slack, Teams, Gmail, and Outlook — the actual mechanism that grows the radius.
  - **Streaming API responses** that render results progressively. A large catalogue shouldn't have to wait on a full payload before the first row appears.
  - **GSAP motion** for reveals and modal choreography — the UI feels considered, not janky.
</CaseBlock>
